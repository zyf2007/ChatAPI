# ChatAPI
[English](EN-README.md) | [简体中文](README.md) | [繁體中文](Zh-Hant-README.md)

[[Telegram](https://t.me/hutao_space)] |  [[LinuxDO](https://linux.do/u/hutao)] | [[BiliBili](https://www.bilibili.com/video/BV11PLg6LEbB)]  
This project allows various AI clients to call a human using the OpenAI Responses style API, and comes with a web console interface to help you assemble Tool Calling requests or set up auto-reply rules.  
Through this project, you can let others configure you as an Agent or chatbot, and then you play the role of an AI assistant being called.
You can also use it as a Mock LLM when developing your own Agent.

- Backend: Flask
- Frontend: React + Vite + Ant Design
- Data storage: SQLite

Default features:

- Login with username/password based on `.env`, optional TOTP support
- Support for three API interfaces: `/v1/chat/completions`, `/v1/responses`, `/messages`
- Conversation list and message persistence for debugging and context inspection
- Automated reply output, supports scheduled streaming, loop output, conditional auto-reply, etc.
- Optional ntfy message push

## 1. Deployment
### One-click deployment without Nginx
#### Build frontend

```bash
cd ./frontend
npm i
npm run build
```

The homepage shows the current access source as the API base URL by default; if you need to specify another base URL at build time, set `VITE_HOMEPAGE_API_BASE_URL` before building.

```env

CHATAPI_PASSWORD=your_password
# Optional; if not provided, the backend will auto-generate and write to the database config table on first start
# CHATAPI_SESSION_SECRET=random_string
```

```bash
cd ./backend
```

## 8. Security enhancement recommendations

#### 8.1 Account security policies

1. Change the default admin password: Modify CHATAPI_PASSWORD immediately after deployment, avoid weak passwords.

2. Enable TOTP two-factor authentication: Enable TOTP in the web console's "System Settings", use with Google Authenticator or other authenticators. The TOTP secret should be stored securely in the database.

3. Enable API Key authentication: Enable Bearer Token authentication for API calls; calls must carry the Authorization: Bearer <api_key> header.

4. Enable rate limiting: Configure request rate limiting in system settings to prevent abuse.

#### 8.2 Network security configuration

1. Use HTTPS in production: Strongly recommended to use Nginx reverse proxy + Let's Encrypt free certificate

2. Restrict CORS origins: Only allow necessary domains, avoid using wildcards

3. Configure firewall: In Termux you can use iptables or nftables to restrict access source IPs

#### 8.3 Sensitive information management

1. After login, enable and save API Key, site title, ntfy address, and TOTP in the web console; these configurations should not be placed in the .env file

2. If SESSION_SECRET is not provided, the backend will auto-generate it on first start and write it to the database config table

## Message Push URL Security Settings

ChatAPI supports sending message notifications via ntfy. Users can enter an ntfy push URL in "My Settings".

### Q: When do I need to modify the "Message Push URL"?

In most cases, leaving it as the default "Disabled" is sufficient. Only enable it when you self-host ntfy on the same machine, same internal network, or a private network as ChatAPI, for example: `http://127.0.0.1:8080/topic` or `http://192.168.1.10:8080/topic`.

If you use the official `https://ntfy.sh/your-topic`, or a self‑hosted ntfy with a public domain name (e.g., `https://ntfy.example.com/topic`), you do not need to change this setting.

Meaning of the three options:

- Disabled: No user may fill in a local or intranet push URL.
- Admin only: Only the administrator may fill in a local or intranet push URL.
- All users: Any logged‑in user may fill in a local or intranet push URL.

It is recommended to use "Admin only" first, and only select "All users" if you fully trust every user. The default is disabled to prevent SSRF risks where users could force the server to access `127.0.0.1`, `localhost`, internal IP addresses, or cloud metadata addresses via the push URL.

## Termux common troubleshooting

###### Q1: Database is locked error

Cause: SQLite may encounter lock conflicts under high concurrency writes.

Solutions:

  1. Enable WAL mode (?journal_mode=WAL)
  
  2. Use connection pooling to limit concurrent connections
  
  3. Wrap busy write operations in transactions

###### Q2: Session Secret related error

Cause: CHATAPI_SESSION_SECRET not set in .env.

Solution: Leave it blank; the backend will auto-generate on first start and write to the database config table. If you need to specify manually, use a high-entropy random string.

###### Q3: Email sending failure

Check items:

  1. SMTP server address and port are correct
  
  2. SMTP authentication is enabled (some providers require it)
  
  3. The sender email has been verified
  
  4. When using third-party APIs (e.g., Resend, Tencent Cloud SES), the API key is valid and has correct permissions. Resend suggests setting permissions to "only send" to reduce leakage risk.

###### Q4: Node.js frontend build fails in Termux

Solutions:
  1. Increase swap space: dd if=/dev/zero of=$PREFIX/swapfile bs=1M count=1024 && chmod 600 $PREFIX/swapfile && mkswap 
  $PREFIX/swapfile && swapon $PREFIX/swapfile
  
  2. Ensure sufficient memory before running npm run build (1GB+ available RAM recommended)

## Nginx reverse proxy example

The following example assumes:

- Frontend static file directory: `/path/to/ChatAPI/frontend/dist`
- Backend address: `http://127.0.0.1:5000`
- Domain name: `chat.example.com`

```nginx
server {
    listen 80;
    server_name chat.example.com;

    root /path/to/ChatAPI/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:5000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /v1/ {
        proxy_pass http://127.0.0.1:5000/v1/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

If you want to enable HTTPS, it is recommended that Nginx handles the certificate rather than using Flask's built-in server.

If you do not want to deploy Nginx additionally, you can also let Flask directly serve both the API and frontend static files:

```env
CHATAPI_WEB_DIST_DIR=./frontend/dist
```

After setting:

- `/api/*` and `/v1/*` continue to go to the backend interface
- Other paths will return static files directly from that directory
- When the requested path does not exist and the directory contains `index.html`, it will automatically fall back to `index.html`, which can be used for frontend single-page application routing
